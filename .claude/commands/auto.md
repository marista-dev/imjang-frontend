# /auto — 전체 자동화 (계획 기반 사이클 반복)

PROGRESS.md의 모든 미완료 Phase를 순서대로 자동 진행해.
각 Phase마다 **계획 → 구현 → 검증 → 리뷰 → 커밋** 사이클을 엄격히 따라.

## 전체 워크플로우

```
 ╔══════════════════════════════════════════════════════════╗
 ║  Phase 루프 (미완료 Phase가 있는 동안 반복)               ║
 ║                                                          ║
 ║  ┌─ 1. PLAN ─────────────────────────────────┐          ║
 ║  │  · PROGRESS.md에서 다음 미완료 Phase 식별    │          ║
 ║  │  · 상세 계획 수립 + PROGRESS.md에 기록       │          ║
 ║  │  · 디자인 체크포인트 정의                    │          ║
 ║  └──────────────────────────────────┬────────┘          ║
 ║                                     ▼                    ║
 ║  ┌─ 2. IMPLEMENT ────────────────────────────┐          ║
 ║  │  · 계획의 항목을 1번부터 순서대로 구현       │          ║
 ║  │  · 계획에 없는 것 추가 금지                  │          ║
 ║  │  · 중간중간 디자인 체크포인트 확인            │          ║
 ║  └──────────────────────────────────┬────────┘          ║
 ║                                     ▼                    ║
 ║  ┌─ 3. VERIFY ───────────────────────────────┐          ║
 ║  │  · npm run build → 에러 시 수정 (최대 3회)  │          ║
 ║  │  · dev server 실행 → HTTP 200 확인          │          ║
 ║  │  · 코드 품질 점검 (console.log, 하드코딩 등) │          ║
 ║  └──────────────────────────────────┬────────┘          ║
 ║                                     ▼                    ║
 ║  ┌─ 4. REVIEW (냉철한 최종 리뷰) ───────────┐          ║
 ║  │  · 최초 계획 vs 구현 결과 1:1 대조          │          ║
 ║  │  · CLAUDE.md 가이드라인 전수 점검            │          ║
 ║  │  · 모바일 / 디자인 / UI 패턴 / 코드 구조    │          ║
 ║  │  · ✅ → 다음 단계 / ❌ → 수정 후 3번 재실행  │          ║
 ║  └──────────────────────────────────┬────────┘          ║
 ║                                     ▼                    ║
 ║  ┌─ 5. COMMIT ───────────────────────────────┐          ║
 ║  │  · git add -A + git commit (한국어 메시지)  │          ║
 ║  │  · git push origin main                    │          ║
 ║  │  · PROGRESS.md 체크 + 커밋 로그 기록        │          ║
 ║  └──────────────────────────────────┬────────┘          ║
 ║                                     ▼                    ║
 ║  ┌─ 6. PROGRESS CHECK ──────────────────────┐          ║
 ║  │  · 다음 미완료 Phase 확인                   │          ║
 ║  │  · 있으면 → 1번으로 돌아가서 반복            │          ║
 ║  │  · 없으면 → 전체 완료 보고                   │          ║
 ║  └───────────────────────────────────────────┘          ║
 ╚══════════════════════════════════════════════════════════╝
```

## 엄격한 규칙

### 절대 생략 금지
- **PLAN 생략 금지**: 계획 없이 구현 시작하지 마.
- **REVIEW 생략 금지**: 리뷰 없이 커밋하지 마.
- **단계 순서 변경 금지**: 반드시 1→2→3→4→5→6 순서.

### 실패 처리
- VERIFY 실패 (빌드 에러): 수정 후 재빌드, 최대 3회. 3회 실패 → 중단 + 보고.
- REVIEW 실패: 수정 후 VERIFY → REVIEW 재실행, 최대 2회. 2회 실패 → 중단 + 보고.
- PUSH 실패: 원인 보고 후 중단.

### 품질 기준
- 빌드가 성공해야 커밋 가능.
- 리뷰가 ✅여야 푸시 가능.
- PROGRESS.md가 업데이트되어야 다음 Phase로 이동 가능.

### 중단 조건
다음 중 하나라도 해당하면 즉시 멈추고 사용자에게 보고해:
- 빌드 실패 3회 연속
- 리뷰 실패 2회 연속
- git push 실패
- 알 수 없는 에러 발생
- 계획과 현저히 다른 결과물이 나온 경우
