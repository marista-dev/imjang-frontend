# /review — 푸시 전 냉철한 최종 리뷰

커밋 & 푸시 전, **최초 계획 대비 결과물**을 냉철하게 검증해.
이 단계는 "내가 만든 코드를 의심하는" 관점에서 진행해.

## 리뷰 순서

### 1단계: 계획 vs 결과 대조
PROGRESS.md에서 해당 Phase의 **최초 계획**을 다시 읽어.
그리고 실제 구현된 코드를 비교해서 다음을 점검해:

- [ ] 계획의 모든 구현 항목이 실제로 구현되었는가?
- [ ] 계획에 없는 불필요한 코드가 추가되지 않았는가?
- [ ] 계획의 API 연동이 정확한 엔드포인트로 되어 있는가?
- [ ] 계획의 예상 커밋 메시지가 실제 변경 내용과 일치하는가?

### 2단계: CLAUDE.md 가이드라인 대조
CLAUDE.md를 다시 읽고, 다음을 하나씩 점검해:

**모바일 퍼스트 (하나라도 위반 시 수정)**
- [ ] 375px 기준 레이아웃이 깨지지 않는가?
- [ ] 앱 컨테이너 max-w-app(430px) 안에 있는가?
- [ ] 터치 타겟 44px × 44px 이상인가? (실제 클래스 확인)
- [ ] input font-size 16px 이상인가? (h-12 + text-base 확인)
- [ ] pb-24로 BottomNav 가림이 방지되는가?
- [ ] safe-area-inset-bottom이 적용되는가?

**디자인 시스템 (하나라도 위반 시 수정)**
- [ ] 하드코딩 색상값(#059669 등)이 코드에 직접 쓰이지 않았는가? → bg-primary, text-primary 등 Tailwind 토큰만
- [ ] 카드: rounded-2xl + shadow-sm + border border-slate-200 + bg-white
- [ ] 버튼: h-12 + rounded-xl + active:scale-[0.98]
- [ ] Input: h-12 + rounded-xl + focus:ring-2 focus:ring-primary/20
- [ ] 폰트: Pretendard가 적용되는 경로인가?

**UI 패턴 (누락 시 추가)**
- [ ] 빈 상태(Empty State) UI가 존재하는가?
- [ ] 로딩 상태(Spinner)가 존재하는가?
- [ ] 에러 상태 처리가 존재하는가?
- [ ] 사용자 액션에 피드백이 있는가? (토스트, 버튼 비활성화 등)

### 3단계: 코드 구조 리뷰
- [ ] 파일이 올바른 디렉토리에 있는가? (pages/, components/, hooks/, api/)
- [ ] 네이밍이 컨벤션을 따르는가? (PascalCase 컴포넌트, camelCase 훅)
- [ ] 불필요한 주석, TODO, console.log가 남아있지 않은가?
- [ ] 중복 코드가 있으면 공통 컴포넌트/훅으로 추출했는가?

### 4단계: 판정
모든 체크를 완료한 후 PROGRESS.md에 리뷰 결과를 기록해:

```markdown
#### 🧊 최종 리뷰 결과 (시각: YYYY-MM-DD HH:MM)

**계획 대비 이행률**: N/N 항목 완료
**가이드라인 준수**: ✅ 모두 충족 / ⚠️ N건 수정 후 충족
**수정 사항**:
  - (수정한 내용이 있으면 기록)
**판정**: ✅ 커밋 & 푸시 승인 / ❌ 재작업 필요
```

#### 판정이 ✅인 경우:
"✅ 최종 리뷰 통과. /commit으로 커밋 & 푸시를 진행하세요." 라고 알려줘.

#### 판정이 ❌인 경우:
- 문제점을 구체적으로 나열해.
- 수정 후 다시 /review를 실행하라고 안내해.
- **절대 리뷰 실패 상태에서 커밋하지 마.**

## 리뷰 태도
- 자기가 만든 코드를 의심해. "됐겠지"가 아니라 "정말 됐나?" 관점.
- 코드를 실제로 열어서 확인해. 기억에 의존하지 마.
- 사소한 것도 놓치지 마 (px-5가 px-4로 되어있다거나, text-slate-500이 아니라 text-gray-500이라거나).
- 가이드라인에 명시된 것은 100% 지켜야 해. "비슷하니까 괜찮겠지"는 안 돼.
